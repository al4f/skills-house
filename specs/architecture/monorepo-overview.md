# Monorepo Overview

**Date:** 2026-07-03  
**Status:** Adopted

## Purpose

This document describes **repository structure and the build pipeline**. For vision and principles, see [framework-vision.md](./framework-vision.md). For install channels and publish workflow, see [distribution.md](./distribution.md).

## Repository layout

```
skills-house/
├── specs/                          # architecture & design docs
├── skills/                         # source skill packages (author here)
│   └── <skill-name>/
│       ├── package.json
│       └── SKILL.md                # builder entry (required)
├── scripts/                        # shared execution packages
│   └── <script-package>/
│       └── package.json            # exports → named namespaces
├── internal-scripts/
│   ├── build/                      # @skills-house/build
│   ├── cli/                        # optional dev CLI (pnpm skills)
│   └── install/                    # install-skills.sh, remove-skills.sh
├── skills-dist/                    # generated Agent Skills output
│   └── <skill-name>/
├── content/publish/                # install, npm, progress guides
├── website/                        # al4f.dev static site
├── pnpm-workspace.yaml
├── package.json
└── .nvmrc
```

## Workspaces

| Workspace | Role | Published |
|-----------|------|-----------|
| `skills/` | Skill authoring. Freeform layout; entry = `SKILL.md`. | Via skills.sh (Git) + optional npm dist |
| `scripts/` | Reusable execution logic referenced via markdown links. | Private monorepo |
| `internal-scripts/build/` | Marker parser, bundler, dist writer. | Published (`@skills-house/build`) |
| `skills-dist/` | Transpiled Agent Skills artifacts. | Packed into npm dist packages |

### `internal-scripts` scope

**build** — marker parser, link resolver, dist writer:

- Parsing `@include` (nested; cycles rejected) and markdown links
- Resolving script namespace exports from `scripts/` packages
- Writing spec-compliant output to `skills-dist/`
- Injecting skill-dependency notes for skill-package links

**install** — `install-skills.sh` / `remove-skills.sh` for monorepo dev and optional CLI packaging.

**cli** — optional `pnpm skills add` wrapper; consumers use the official skills.sh CLI instead (see [distribution.md](./distribution.md)).

## Source vs dist

### Source (`skills/`)

- Layout is **open** — authors are not forced into the Agent Skills folder tree.
- **Entry:** `SKILL.md`
- **`@include /path`** — only build marker; nested includes supported.
- **Markdown links** `[label](target)` — in-package files (`/path`) or package refs (`package/export`).

Authoring rules: [SKILL.md authoring spec](../authoring/skill-md-authoring.md). Frontmatter fields: [skill-frontmatter.md](../schema/skill-frontmatter.md).

### Dist (`skills-dist/`)

- Pure build output — **do not validate**.
- Conforms to [Agent Skills directory structure](https://agentskills.io).
- Generated by `pnpm build`; used for local install and npm dist publish.

| Directory | Purpose |
|-----------|---------|
| `references/` | Documentation loaded on demand |
| `scripts/` | Executable code (bundled or copied) |
| `assets/` | Static resources |

## Build pipeline

Package: `@skills-house/build` in `internal-scripts/build/`.

### Commands

```bash
pnpm build                                    # compiler + skills
pnpm --filter @skills-house/skill-auditor build
node internal-scripts/build/dist/cli.js skills/skill-auditor   # CI-style single skill
```

### Build steps (per skill)

1. **Discover** — `skills/<name>/SKILL.md` + `package.json`
2. **Parse** — `@include` markers and markdown links in body
3. **Resolve `@include`** — recursive merge; reject cycles
4. **Resolve in-package links** — copy to dist; rewrite hrefs
5. **Resolve package links** — bundle scripts or inject skill-dependency notes
6. **Write output** — `skills-dist/<name>/` with valid frontmatter
7. **Done** — no validation of dist output

## Validation

- Each package defines its own `validate` script.
- **`skills-dist/` is never validated** — generated output only.

```bash
pnpm validate
```

## Skill-to-skill references

When the builder encounters `[other-skill](other-skill)`:

1. Record dependency in frontmatter metadata.
2. Replace with an install-suggestion note (format defined in [distribution.md](./distribution.md#skill-dependency-notes-build-output)).
3. No file copy.
